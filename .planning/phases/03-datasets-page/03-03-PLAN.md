**Part 1: CRUD Verification**

Run the backend and frontend dev servers, then verify the following via curl or the browser. The goal is to confirm the existing CRUD pipeline works -- no code changes expected, only verification.

1. **List datasets**: `curl http://localhost:8000/api/datasets/managed` -- should return a JSON array (possibly empty if no datasets exist yet). Verify the response is 200 and the structure matches `RecvizDataset[]`.

2. **Create a dataset**: POST to `/api/datasets/managed` with:
```json
{
  "name": "Test Dataset",
  "description": "Phase 3 verification test",
  "database_id": "<ID of an existing database from /api/databases>",
  "sql": "SELECT * FROM dual WHERE 1={{filters}}",
  "columns": []
}
```
Verify: 201 response with the created dataset object including a generated `id`.

3. **Get dataset**: `curl http://localhost:8000/api/datasets/managed/<id>` -- verify 200 with the dataset object.

4. **Update dataset**: PUT to `/api/datasets/managed/<id>` with:
```json
{
  "name": "Updated Test Dataset",
  "description": "Updated description"
}
```
Verify: 200 response with updated name/description.

5. **Execute query**: POST to the SQL execution endpoint (check exact path in `use-sql-execute.ts`):
```json
{
  "sql": "SELECT 1 AS test_col, SYSDATE AS now FROM dual",
  "database_id": "<ID>",
  "limit": 100
}
```
Verify: 200 response with `status: "success"`, `columns` array with column metadata, `data` array with at least 1 row, `rowCount` >= 1.

6. **Delete dataset**: DELETE to `/api/datasets/managed/<id>`.
Verify: 200 or 204 response, subsequent GET returns 404.

7. **Parameterized SQL**: If `_build_sql()` in `query_engine.py` resolves `{{filters}}`, `{{values}}`, `{{date_range_clause}}` placeholders, verify that a query with these placeholders executes without error when the placeholders are replaced with defaults (e.g., `1=1`).

If any endpoint returns an error, investigate and fix the issue. Common issues:
- Missing database ID (verify databases exist first via `GET /api/databases`)
- Sync vs async handler mismatch (all handlers must be `def`, not `async def`)
- Oracle connection issues (verify backend is running with thick mode)

# Phase 02.1: Chart Rendering Foundation - Research

**Researched:** 2026-04-05
**Domain:** AG Charts Enterprise / ECharts series mapping, config-driven chart rendering, Playwright E2E testing
**Confidence:** HIGH

## Summary

Phase 2.1 fixes a fundamental bug in the chart rendering pipeline: `buildSeries()` in `ag-chart-wrapper.tsx` blindly uses `columns[0]` as category and `columns.slice(1)` as metrics, ignoring the config metadata that specifies which columns are metrics. This means every chart type "works by accident" only when column order happens to match expectations. The fix is config-driven: expand `ChartConfig` with `metricColumns` and `categoryColumn`, populate them from `DashboardChartConfig.sources[].metric` in `config-chart-grid.tsx`, and refactor `buildSeries()` to use these fields.

Additionally, heatmap and treemap chart types are not implemented in `buildSeries()` at all (the `default` case silently renders them as bar charts). The waterfall chart is also incorrectly implemented as a plain bar instead of using AG Charts' native `waterfall` series type. ECharts wrapper needs the same config-driven audit. A showcase dashboard with per-chart-type data sources provides the comprehensive verification that every chart type renders real data correctly. Playwright E2E tests validate both chart rendering and Phase 2 cross-filter/drill-down interactions visually.

**Primary recommendation:** Refactor `buildSeries()` to accept `metricColumns`/`categoryColumn` from config, add heatmap/treemap/waterfall native series types, create showcase dashboard with dedicated seed data per chart type, and validate everything with Playwright E2E tests.

### Pattern 1: Config-Driven Column Resolution
**What:** `buildSeries()` receives explicit metric/category column names from config instead of deriving from column order.
**When to use:** Every chart render path.
**Example:**
```typescript
// Source: D-01, D-02 from CONTEXT.md

// BEFORE (broken -- column order dependent):
function buildSeries(vizType: string, columns: string[], selection?: ChartSelection) {
  const categoryKey = columns[0] ?? 'category'
  const metricKeys = columns.slice(1)
  // ...
}

// AFTER (config-driven):
function buildSeries(
  vizType: string,
  columns: string[],
  metricColumns: string[],
  categoryColumn: string | undefined,
  selection?: ChartSelection,
) {
  // Resolve category: explicit config > first non-metric string column
  const categoryKey = categoryColumn
    ?? columns.find((c) => !metricColumns.includes(c))
    ?? columns[0]
    ?? 'category'
  const metricKeys = metricColumns.length > 0
    ? metricColumns
    : columns.filter((c) => c !== categoryKey)
  // ...
}
```

### Anti-Patterns to Avoid
- **Column order assumption:** `columns[0]` as category, `columns.slice(1)` as metrics. This is the core bug being fixed.
- **Silent fallback to bar:** The current `default` case in `buildSeries()` renders unknown types as bar. Financial data platform must not silently misrepresent data.
- **Hardcoded column names:** Never use string literals like `'category'` or `'count'` as default column keys. If config doesn't specify, use the first available column with appropriate type detection.

## ECharts Config-Driven Audit

The ECharts wrapper (`echart-wrapper.tsx`) has the same column-order assumption problem. `buildEChartsOption()` uses `columns[0]` as category and `columns[1]` as metric everywhere. Since ECharts handles 7 chart types (sankey, radar, sunburst, gauge, funnel, graph, parallel), all need the same config-driven treatment.

However, ECharts types have more diverse data shape requirements:
- **Sankey/Graph:** Need `source`, `target`, `value` columns (3 specific roles)
- **Radar:** Category column + multiple metric columns
- **Sunburst:** Hierarchical data (similar to treemap)
- **Gauge:** Single metric value
- **Funnel:** Category + single metric
- **Parallel:** All columns treated as dimensions

**Recommendation:** For ECharts, the `metricColumns`/`categoryColumn` pattern works for funnel, radar, and gauge. For sankey/graph, the columns have specific roles (source, target, value) that could be mapped via `params` or positional convention. The current positional approach (columns[0]=source, columns[1]=target, columns[2]=value) is actually reasonable for these types since the data source schema is designed specifically for the chart type. The config-driven fix should focus on types where column order ambiguity causes bugs (funnel, gauge, radar).

## Common Pitfalls

### Pitfall 1: Treemap Hierarchical Data Requirement
**What goes wrong:** AG Charts treemap expects `children` arrays in data. Passing flat rows renders nothing or crashes.
**Why it happens:** Backend returns flat rows from SQL queries. Treemap needs transformation.
**How to avoid:** Add a `flatToHierarchy()` utility in the chart wrapper that groups flat rows by a category column and nests them as children. Apply this transformation specifically for treemap vizType before passing data to AG Charts.
**Warning signs:** Empty treemap, console error about missing children property.
[CITED: https://www.ag-grid.com/charts/react/treemap-series/]

### Pitfall 2: Heatmap Needs Two Category Axes
**What goes wrong:** Heatmap uses xKey AND yKey as categories (both strings), plus colorKey as the numeric value. If you treat it like bar/line (one category, one metric), the chart won't render.
**Why it happens:** Most chart types use one category axis. Heatmap is a matrix with two.
**How to avoid:** For heatmap, the column mapping needs special handling: first column = xKey, second column = yKey, metric column = colorKey. This is a three-column requirement, not the standard two.
**Warning signs:** Heatmap renders as single row/column, or shows error about missing keys.
[CITED: https://www.ag-grid.com/charts/react/heatmap-series/]

### Pitfall 3: Waterfall Type Must Be 'waterfall', Not 'bar'
**What goes wrong:** Current code renders waterfall as `type: 'bar'`, losing cumulative running total visualization, connector lines, and positive/negative coloring.
**Why it happens:** When waterfall was first implemented, the developer may have used bar as a placeholder.
**How to avoid:** Use AG Charts native `type: 'waterfall'` series. This requires Enterprise license (already in use).
**Warning signs:** Waterfall chart looks like a regular bar chart with no connectors or running totals.
[CITED: https://www.ag-grid.com/charts/react/waterfall-series/]

### Pitfall 4: AG Charts Series Type String Must Be Exact
**What goes wrong:** AG Charts silently ignores series with unrecognized type strings. Using `'stacked-bar'` as a type string won't work -- it must be `'bar'` with `stacked: true`.
**Why it happens:** The vizType from config (e.g., `'stacked-bar'`) doesn't map 1:1 to AG Charts series type strings.
**How to avoid:** `buildSeries()` must translate vizTypes to AG Charts types. `'stacked-bar'` -> `type: 'bar'` + `stacked: true`. `'donut'` -> `type: 'donut'`. `'histogram'` -> `type: 'bar'`.
**Warning signs:** Chart renders empty or as wrong type.
[VERIFIED: current mapping in ag-chart-wrapper.tsx handles this correctly for bar/stacked-bar]

### Pitfall 5: Scatter Chart Has No Category Column
**What goes wrong:** Applying the standard category/metric pattern to scatter breaks it. Scatter uses two numeric columns for x/y axes.
**Why it happens:** Config-driven mapping assumes one category + N metrics. Scatter is the exception.
**How to avoid:** Special-case scatter in `buildSeries()`: first two columns (or first two metricColumns) become xKey/yKey. No categoryColumn used.
**Warning signs:** Scatter shows category strings on x-axis instead of numeric values.
[VERIFIED: current scatter implementation at ag-chart-wrapper.tsx lines 99-107]

### Pitfall 6: Cross-Filter categoryKey Must Match Config
**What goes wrong:** `makeItemStyler()` receives the wrong `categoryKey`, so clicking a chart segment cross-filters on the wrong column.
**Why it happens:** categoryKey was derived from `columns[0]` (order-dependent). After fix, must come from resolved config.
**How to avoid:** The categoryKey passed to `makeItemStyler()` and to click event handlers must use the same resolved categoryColumn from config.
**Warning signs:** Cross-filter badge shows wrong column name, dimming doesn't work, or filtering applies wrong values.
[VERIFIED: makeItemStyler at ag-chart-wrapper.tsx lines 24-36, click handler at lines 202-216]

### Pitfall 7: Playwright Tests Need Running Backend+Frontend
**What goes wrong:** Playwright E2E tests require the full stack running (PostgreSQL, Superset, FastAPI, Vite dev server). Tests fail with connection errors if any service is down.
**Why it happens:** E2E tests hit real HTTP endpoints, not mocks.
**How to avoid:** Playwright config should document the required services. Consider a `webServer` config in `playwright.config.ts` to auto-start the frontend dev server. Backend/Superset must be started manually or via docker-compose.
**Warning signs:** Tests timeout, connection refused errors.
[ASSUMED]

### Pitfall 8: Column Normalization Still Required
**What goes wrong:** Superset returns column objects `{column_name, name, type}` instead of strings. If normalization is missed, column matching fails.
**Why it happens:** Phase 2 decision to normalize columns (STATE.md: "Normalize column objects to strings in ChartDataResponse").
**How to avoid:** The column normalization in `config-chart-grid.tsx` (lines 83-85) must remain intact. Metric column matching should compare against normalized string column names.
**Warning signs:** Config says metric is `"total_items"` but normalized column is something different.
[VERIFIED: normalization at config-chart-grid.tsx lines 83-85]

### AG Charts Treemap with Flat-to-Hierarchy Transform
```typescript
// Source: https://www.ag-grid.com/charts/react/treemap-series/
// Flat data must be transformed to hierarchical for treemap

function flatToHierarchy(
  rows: Record<string, unknown>[],
  labelKey: string,
  sizeKey: string,
  colorKey?: string,
): unknown[] {
  // Group rows by labelKey value
  return rows.map((row) => ({
    [labelKey]: row[labelKey],
    [sizeKey]: Number(row[sizeKey] ?? 0),
    ...(colorKey ? { [colorKey]: Number(row[colorKey] ?? 0) } : {}),
  }))
}

case 'treemap': {
  const labelKey = categoryKey
  const sizeKey = metricKeys[0] ?? 'value'
  const colorKey = metricKeys[1]
  return [{
    type: 'treemap' as const,
    labelKey,
```

## Open Questions

1. **Treemap data structure: flat or hierarchical?**
   - What we know: AG Charts treemap documentation shows hierarchical data with `children` arrays. SQL queries return flat rows.
   - What's unclear: Whether AG Charts can render a simple flat array of objects as leaf-only treemap without wrapping in a parent.
   - Recommendation: Plan for a `flatToHierarchy()` transform. If flat works directly, the transform becomes a no-op wrapper. LOW risk either way.

2. **Heatmap three-column mapping approach**
   - What we know: Heatmap needs xKey (cat), yKey (cat), colorKey (numeric). Standard config has one `categoryColumn` and N `metricColumns`.
   - What's unclear: Best way to specify the second category column for heatmap specifically.
   - Recommendation: For heatmap, resolve yKey as the second non-metric column in the data. The data source is designed specifically for heatmap, so column position is predictable. Could also use `params` for explicit override.

3. **ECharts sunburst hierarchical data**
   - What we know: Sunburst requires hierarchical data similar to treemap. Current implementation casts `data as unknown[]` hoping the backend returns nested objects.
   - What's unclear: Whether the showcase dashboard should include a sunburst test or defer to later phases.
   - Recommendation: Include at least one ECharts type (funnel is simplest) in showcase for validation. Sunburst/sankey have complex data requirements that may be deferred if not blocking.

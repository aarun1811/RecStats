# Phase 2.1: Chart Rendering Foundation - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix all major chart types to render correctly from query data using configured metric and category columns. Add heatmap and treemap support to the AG Charts wrapper. Validate Phase 2 cross-filter and drill-down end-to-end with properly rendering charts. Unsupported chart types show explicit error messages instead of rendering incorrectly.

</domain>

<decisions>
## Implementation Decisions

### Column Mapping Strategy
- **D-01:** Config-driven column mapping. The chart wrapper determines metric vs category columns from `DashboardChartConfig.sources[].metric`, NOT from query response column order. First non-metric string column becomes the category key.
- **D-02:** Expand `ChartConfig` type with `metricColumns: string[]` and `categoryColumn?: string`. These are populated from `DashboardChartConfig.sources[].metric` in `config-chart-grid.tsx` and passed through to the chart wrapper. `buildSeries()` uses these instead of `columns[0]`/`columns.slice(1)`.

### Chart Type Coverage
- **D-03:** Full AG Charts support for heatmap and treemap in this phase. Heatmap needs xKey, yKey, colorKey. Treemap needs labelKey, sizeKey, optional colorKey. Both added to `buildSeries()`.
- **D-04:** After this phase, 18 chart types are functional:
  - **AG Charts (11):** bar, stacked-bar, line, area, pie, donut, scatter, heatmap, treemap, waterfall, combo (histogram rendered as bar)
  - **ECharts (7):** sankey, sunburst, radar, gauge, funnel, graph/network, parallel coordinates
- **D-05:** Unsupported chart types (e.g., bullet, box-plot) show an explicit inline error panel: "Unsupported chart type: X" in the chart card. No silent bar chart fallback. These types get added when a dashboard actually needs them (Phase 6: Chart Library).

### Visual Validation
- **D-06:** Create a new purpose-built "chart-showcase" test dashboard config with one of each chart type (bar, line, area, pie, donut, scatter, heatmap, treemap), cross-filter enabled, and drill hierarchies defined. Seed data designed to look good in all chart types.
- **D-07:** Playwright E2E tests for key interaction paths: click chart element to cross-filter (assert dimming + badge bar), double-click to drill (assert breadcrumb + detail grid), plus basic render tests for each chart type.
- **D-08:** Manual validation against running app with the showcase dashboard as the final verification step.

### Error Resilience
- **D-09:** When a config-specified metric column is missing from query data, show an inline warning panel in the chart card: "Column 'X' not found in data source response. Available: [list]". No chart rendered — no attempt to fall back to wrong data. Clear, actionable message for config authors.
- **D-10:** Same approach for missing category column — explicit error, not silent degradation. Financial data platform cannot silently misrepresent data.

### Claude's Discretion
- Heatmap/treemap series builder implementation details (colorRange, aggregation keys)
- Playwright test structure and assertion patterns
- Showcase dashboard seed data content and schema
- How `ChartConfig.categoryColumn` is resolved when not explicitly set (first non-metric string column detection logic)
- EChart wrapper adjustments if needed for config-driven column mapping

</decisions>

<specifics>
## Specific Ideas

- The core bug is that `buildSeries()` ignores `sources[].metric` and blindly uses column order. The fix is config-driven: metric columns come from config, category is resolved as first non-metric string column.
- Heatmap and treemap are AG Charts Enterprise types that need specific data key mapping (heatmap: xKey+yKey+colorKey, treemap: labelKey+sizeKey+colorKey).
- Unsupported chart types must show explicit error, not silent fallback. This is a financial data platform -- misrepresenting data via wrong chart type is dangerous.
- The Playwright tests serve dual purpose: validate Phase 2 cross-filter/drill-down AND provide regression safety for chart rendering going forward.
- The showcase dashboard is a testing tool, not a production dashboard. Seed data should be synthetic but realistic enough to validate all chart types.

</specifics>

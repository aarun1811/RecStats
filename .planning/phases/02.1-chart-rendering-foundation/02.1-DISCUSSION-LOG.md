## Column Mapping Strategy

### Q1: How should the chart wrapper know which columns are metrics vs categories?

| Option | Description | Selected |
|--------|-------------|----------|
| Config-driven (Recommended) | Pass sources[].metric from DashboardChartConfig through to the chart wrapper. Metric columns come from config; first non-metric string column becomes category. Explicit, predictable, matches how configs are authored. | ✓ |
| Hybrid: config + fallback | Use sources[].metric when available, but fall back to current heuristic (columns[0] = category, rest = metrics) when config lacks metric metadata. More forgiving of incomplete configs. | |
| Query-order heuristic | Keep current approach but improve it -- detect string columns as categories, numeric columns as metrics via runtime type inspection. No config dependency, but fragile with mixed-type data. | |

**User's choice:** Config-driven (Recommended)
**Notes:** None

### Q2: Should the chart wrapper receive column mapping as explicit props, or should ChartConfig be expanded?

| Option | Description | Selected |
|--------|-------------|----------|
| Expand ChartConfig | Add metricColumns: string[] and categoryColumn?: string to ChartConfig type. Chart wrapper uses these directly. Single source of truth, cleaner API. | ✓ |
| Separate props | Add metricColumns and categoryColumn as new optional props on ChartWrapperProps. Keeps ChartConfig unchanged. More flexible but more surface area. | |
| You decide | Claude chooses the cleanest approach based on existing patterns and minimal disruption. | |

**User's choice:** Expand ChartConfig
**Notes:** None

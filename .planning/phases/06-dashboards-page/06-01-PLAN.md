<objective>
Fix the broken dashboard data pipeline by rewiring ConfigStore from the Superset-era `recviz_data_sources` table to `recviz_datasets` + `recviz_connections`.

Purpose: This is the critical blocker -- dashboards built from user-created datasets currently 404 at render time because `recviz_data_sources` is only populated by the seed script. Without this fix, no dashboard verification is possible.

Output: ConfigStore reads from `recviz_datasets`, builds `DataSourceConfig` from dataset fields, resolves connection names from `recviz_connections`. All `/api/data-sources/*` endpoints work for both seeded and user-created datasets.
</objective>

The dashboard builder stores `item.chart.datasetId` as `sources[].dataSourceId` in the dashboard config. So when the renderer calls `/api/data-sources/{dataSourceId}/query`, the `dataSourceId` is actually a `recviz_datasets.id`.

The seed script currently creates PAIRED rows: one `recviz_datasets` row and one `recviz_data_sources` row with the SAME ID. The `recviz_data_sources.config` JSON contains `{ id, name, database_routing: { type: "static", database: "<connection_name>" }, query: "<sql>", columns: [...], filter_mappings: [...] }`.

The fix: ConfigStore reads `RecvizDataset` by ID, resolves the connection name from `RecvizConnection` using `database_id`, and builds a `DataSourceConfig` from the dataset fields.

  <action>
Rewrite `ConfigStore` (per D-01, D-02) to read from `RecvizDataset` and `RecvizConnection` instead of `RecvizDataSource`:

1. **`config_store.py`**: Replace the `RecvizDataSource` import with `RecvizDataset` and `RecvizConnection`. Remove the `config_migrator` import (no longer needed for this path).

2. **`get_data_source(self, data_source_id: str)`**:
   - Query `RecvizDataset` by ID: `self._session.get(RecvizDataset, data_source_id)`
   - If not found, return None
   - Look up the connection: `self._session.execute(select(RecvizConnection).where(RecvizConnection.id == dataset.database_id))` to get the connection name
   - Build a `DataSourceConfig` from the dataset fields:
     - `id` = dataset.id
     - `name` = dataset.name
     - `database_routing` = `DatabaseRoutingMapping(type="static", database=connection.name)` (the connection NAME, not ID, because `ConnectionResolver.resolve()` maps names to UUIDs)
     - `query` = dataset.sql
     - `columns` = build `ColumnDef` list from dataset.columns JSON (each column has `name`, `type` fields; map `role`->`type` if needed, default type to "string")
     - `filter_mappings` = build from dashboard config's filter definitions OR from the dataset's column metadata. For now, default to empty list `[]` -- filter mappings are stored in the dashboard config, not the dataset. The `_build_sql` in QueryExecutor handles `{{filters}}` template replacement using whatever filter_mappings are provided. Dashboard filter configs map filter IDs to SQL expressions, and these are passed through the dashboard renderer's query calls.

3. **`list_data_sources(self)`**: Same pattern -- query all `RecvizDataset` rows, join/lookup connections, build `DataSourceConfig` list.

4. **`data_source_config.py`**: No structural changes needed. The `DataSourceConfig` model already has the right shape. The `DatabaseRoutingMapping` with `type="static"` and `database=<connection_name>` is exactly what `QueryExecutor._resolve_database()` expects.

IMPORTANT: The `RecvizDataset.columns` field stores a JSON array where each element has at minimum `{ name: str, type: str }` matching `ColumnDef`. Some may also have `role`, `aggregation`, `format` fields from the dataset editor -- ignore those extra fields, only extract `name`, `type`, and optionally `label`.

IMPORTANT: When the connection lookup fails (dataset.database_id doesn't match any RecvizConnection.id), log a warning and return None rather than crashing -- this handles orphaned datasets gracefully.

## Interaction Contract

### Share button

| Property | Value |
|----------|-------|
| Component | `<ShareLinkButton />` (new) |
| Visual | `<Button variant="outline" size="sm"><Share2 className="mr-1.5 size-4" />Share</Button>` wrapped in `<Tooltip>` |
| Tooltip delay | 300ms (Shadcn Tooltip default + CLAUDE.md animation rule) |
| Click action | `await navigator.clipboard.writeText(window.location.href)` |
| Success feedback | `toast.success('Link copied')` â€” Sonner default duration, no custom variant |
| Failure feedback | `toast.error('Could not copy link')` â€” wrapped in try/catch around `writeText` |
| Keyboard access | Tab-focusable, Enter/Space activates (native `<button>` behavior) |
| Disabled state | None. Button is always enabled (no prerequisite state). If clipboard API rejects, the error toast fires. |
| Loading state | None. Copy is synchronous from user perspective (sub-millisecond clipboard write). |
| Icon choice | `Share2` from lucide-react. **Rationale:** `Share2` is the standard share glyph in lucide (two dots connected by lines â€” universally understood as "share"). `Link` would read as "hyperlink inserter" and `ExternalLink` is already taken by the EmbedTopbar's "Open in RecViz" action. Lock: `Share2`. |
| Placement | Inside the header `flex items-center justify-between` cluster on `routes/_app/dashboards/$dashboardId.tsx`, immediately before the existing Edit button, wrapped in a `div.flex.items-center.gap-2` to provide the sibling spacing. |
| Only in view mode? | Yes. The edit mode (`$dashboardId.edit.tsx`) does not render a Share button â€” sharing a dashboard that is actively being edited would expose in-progress state. |

### EmbedTopbar hide tokens

| Token | Effect on EmbedTopbar | Effect on DashboardRenderer |
|-------|-----------------------|------------------------------|
| (no `?hide=` param) | Title visible, "Open in RecViz" link visible | `ConfigFilterBar` visible, `DashboardToolbar` visible |
| `?hide=title` | Title `<span>` is not rendered. Bar still has the `h-9` height and `border-b` separator. The "Open in RecViz" link remains visible (right-aligned, the only element in the bar). | Unchanged |
| `?hide=filter-bar` | Unchanged (title + link still visible) | `ConfigFilterBar` is not mounted |
| `?hide=toolbar` | Unchanged | `DashboardToolbar` (refresh + auto-refresh) is not mounted |
| `?hide=filter-bar,title,toolbar` | Only "Open in RecViz" link remains in the topbar | Neither `ConfigFilterBar` nor `DashboardToolbar` is mounted |

**Important semantic clarification (locked here so the planner does not re-debate it):**

- `?hide=toolbar` hides the `DashboardToolbar` (inside `DashboardRenderer`), NOT the `EmbedTopbar`. The EmbedTopbar always remains visible in Phase 9 (per D-06).
- `?hide=title` hides ONLY the title text inside `EmbedTopbar`, not the whole topbar.
- There is no token in Phase 9 that hides the entire `EmbedTopbar`. Full chromeless mode is explicitly out of scope.

**Visual transitions:** None. Hide tokens are evaluated on mount (from URL). They do not animate in/out. Changing the URL tokens requires a page reload (embed mode â€” no client-side re-parsing of `?hide=`).

### URL state sync (Share button's underlying behavior)

| Property | Value |
|----------|-------|
| Read timing | On mount (`useEffect` on `[dashboardId]`) â€” URL params â†’ `filterStore.applyFilters(initial)` via the renderer's `initialFilters` prop |

| History mode | `replace: true` â€” URL updates in place, back button is not polluted with intermediate filter states |

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Executor swapped hooks without understanding api-client transforms | Executor replaced `useDashboardConfig` (flat endpoint) with `useManagedDashboard` (wrapped endpoint) without realizing that `config` is in `DATA_KEYS`, so snake_case keys inside stay un-transformed. Broke `DashboardRenderer` which reads camelCase. | advisory | Before swapping data-fetching hooks, verify the response shape matches what consumers expect. Check api-client.ts DATA_KEYS set and how the receiving component reads fields. |
| Cross-entity ID confusion (dataset UUID vs legacy data source ID) | `BuilderChartContent` called `useDataSourceQuery(chart.datasetId)` which hits `/api/data-sources/{id}/query` тАФ that endpoint expects legacy config data source IDs, not managed dataset UUIDs. All chart adds in builder returned 404. | advisory | Two different "data source" systems exist: legacy `recviz_data_sources` (config-based, used by tlm-stats/chart-showcase dashboards) and managed `recviz_datasets` (SQL execute via /api/sql/execute). Check which system the ID belongs to before calling an endpoint. |
| Orphaned components after plan execution | `DeleteDashboardDialog` was built in plan 08-09 but never wired into `DashboardList`. The verifier caught it. | advisory | When a component is listed in `files_modified`, verify it's actually imported and rendered somewhere in the integration step of the plan. |

<decisions_made>

- **Dashboard view/edit routes use flat endpoint**: `useDashboardConfig` calls
  `/api/dashboards/:id` (returns flat `DashboardConfig` with proper camelCase
  transform) instead of `useManagedDashboard` which wraps config in a
  `DATA_KEYS`-skipped field. Both endpoints read the same recviz_dashboards
  table so user-created dashboards still work.

- **Builder chart content uses SQL execute**: `BuilderChartContent` fetches
  the managed chart via `useManagedChart` to get `columnMapping`, fetches
  the managed dataset via `useManagedDataset` to get SQL, then executes via
  `/api/sql/execute`. Same pattern as `chart-library-card.tsx` and
  `chart-detail-panel.tsx`. KPI and grid panels also use this pattern via
  a shared `useDatasetPreview` helper.

- **Library aesthetic = refined minimalism with signature art**: Each
  dashboard card gets a unique flowing bezier curve derived from
  `FNV-1a(dashboard.id)`. 5 anchor points, smooth Catmull-Rom-to-bezier
  path, gradient fill + gradient stroke + glow halo + data point dots.
  Soft ambient radial gradients + subtle grain overlay. Feels like album
  art for dashboards.

- **Builder aesthetic = architectural blueprint**: Dot grid canvas (primary/12),
  architectural corner ticks, `12-COL GRID` mono caption, `DSH / NEW`
  monospace document-ID label above the title, distinctive SVG blueprint
  illustration for empty state (KPI row + area chart + bar chart + table
  with corner marks). `EDIT MODE` / `UNSAVED EDITS` mono mode indicator
  with glowing dot. Two modes, two moods тАФ library is calm product gallery,
  builder is creative workspace.

- **BuilderPanel has type-specific icons + mono badges**: `BarChart3` + `CHART`,
  `Gauge` + `KPI`, `Table2` + `GRID`. Opaque `bg-card` header (was
  `bg-muted/20` тАФ caused canvas text bleed-through).

</decisions_made>

1. Create `frontend/src/components/builder/filter-column-mapper.tsx` (D-21):

```typescript
interface ChartColumnMapping {
  chartItemId: string
  chartTitle: string
  datasetId: string
  matchedColumn: string | null  // null if no auto-match
  availableColumns: string[]    // all columns from this chart's dataset
}

interface FilterColumnMapperProps {
  filterColumnName: string  // The column name being used as filter
  chartMappings: ChartColumnMapping[]
  onMappingChange: (chartItemId: string, mappedColumn: string | null) => void
}
```

This component shows which charts auto-matched the filter column and allows manual override:
- For each chart on the dashboard, check if its dataset has a column with the same name as the filter column
- If yes: show a green checkmark + "Auto-matched: {column}" — user can override via dropdown
- If no match: show an amber warning + "No match" — user can pick a column from that chart's dataset columns to map, or leave unmapped (chart is excluded from this filter)
- Unmatched charts are excluded from filtering (they don't receive the WHERE clause for this filter)

UI layout:
```tsx
<div className="space-y-2 mt-3">
  <Label className="text-xs font-medium text-muted-foreground uppercase">Column Mapping Per Chart</Label>
  <p className="text-xs text-muted-foreground">Charts with matching column names are auto-mapped. Override for charts with different column names.</p>
  {chartMappings.map(mapping => (
    <div key={mapping.chartItemId} className="flex items-center justify-between gap-2 rounded border px-2 py-1.5">
      <div className="flex items-center gap-2 min-w-0">
        {mapping.matchedColumn ? (
          <Check className="size-3 text-green-600 dark:text-green-400 shrink-0" />
        ) : (
          <AlertTriangle className="size-3 text-amber-600 dark:text-amber-400 shrink-0" />
        )}
        <span className="text-xs truncate">{mapping.chartTitle}</span>
      </div>
      <Select
        value={mapping.matchedColumn ?? '__none__'}
        onValueChange={(v) => onMappingChange(mapping.chartItemId, v === '__none__' ? null : v)}
      >
        <SelectTrigger className="h-7 text-xs w-[140px]">
          <SelectValue placeholder="No match" />
        </SelectTrigger>
        <SelectContent>
          <SelectItem value="__none__">Exclude</SelectItem>
          {mapping.availableColumns.map(col => (
            <SelectItem key={col} value={col}>{col}</SelectItem>
          ))}
        </SelectContent>
      </Select>
    </div>
  ))}
</div>
```

   - Auto-detected filter type badge:
     - `dataType === 'string'` or `role === 'dimension'` -> `<Badge variant="secondary">Multi Select</Badge>`
     - `dataType === 'date'` or `role === 'time'` -> `<Badge variant="secondary">Date Range</Badge>` (maps to 'preset-range' FilterConfig type)
     - `dataType === 'number'` or `dataType === 'currency'` and `role === 'measure'` -> `<Badge variant="secondary">Number Range</Badge>` (use 'preset-range' type)
   - Override dropdown: `<Select>` to change filter type (Single Select, Multi Select, Date Range, Number Range)
   - Each selected column gets an optional "Depends on" dropdown (D-22): lists other already-selected filter columns. Maps to `FilterOptionsSource.dependsOn`.

d) **Column mapping section (D-21) — shown when a column is selected**: For each selected column, show a `<FilterColumnMapper>` below it:
   - Build `chartMappings` by scanning all BuilderItems (charts and grids) on the dashboard
   - For each chart/grid item, check if its dataset has a column with the same name as the selected filter column
   - Auto-match: if column name exists in the dataset, set `matchedColumn` to that column name
   - No match: set `matchedColumn` to null
   - Fetch each item's dataset columns via `useManagedDatasets()` (already loaded above)
   - The user can override mappings via the FilterColumnMapper dropdowns
   - Store the mapping in local state; when "Add Filters" is clicked, encode the mappings into the FilterConfig

e) **On "Add Filters" click**: Build `FilterConfig[]` from selected columns:
   - `id`: `crypto.randomUUID()`
   - `label`: column `displayName` or `name`
   - `type`: auto-detected or user-overridden ('single-select' | 'multi-select' | 'preset-range')
   - `lockable`: false
   - `optionsSource`: `{ dataSourceId: dataset.id (per Research Item 1 — first dataset containing the column), valueColumn: column.name, dependsOn: {} or { parentFilterId: parentColumnName } }`
   - Note: The column mapping per chart is stored as metadata that the cross-filter system uses at runtime. The existing cross-filter system uses column-name matching. For charts where the user mapped to a different column name, the builder should store this in the DashboardChartConfig or as extended FilterConfig metadata. For now, store column mappings in a `columnMappings` property on the FilterConfig (extend the type if needed) or document that cross-filter column mapping override requires a DashboardConfig extension (Phase 2 D-07 uses column-name matching only — see RESEARCH.md Research Item 2). The key deliverable is the UI for manual mapping; the runtime wiring can be extended in the DashboardConfig type as needed.
   Call `onFiltersAdded(filters)`.

f) **Skip columns already added as filters**: Check existing `filters` in builder store, grey out or hide columns that already have a matching filter.

g) **UI layout per UI-SPEC**: Dialog max-w-lg, ScrollArea, footer with [Add Filters] button.
